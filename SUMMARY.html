The Flaky Network

Architecture & Design Choices — Real-Time UDP Stream Recovery

1. Problem Statement

A source produces one 160-byte frame every 20 ms (simulating live audio). Frames must cross a hostile UDP relay that drops, delays, reorders, and duplicates packets, and be played out on a steady 20 ms clock at the receiver. We build the sender and receiver programs that sit on either side of this relay.

3. Wire Protocol Design

All sender→receiver packets share a 5-byte header (type + 4-byte big-endian sequence/ID), followed by 160 bytes of payload, totaling 165 bytes. This minimal header was chosen to conserve the 2.0× bandwidth budget.

4. Loss Recovery: XOR FEC

Why FEC Over Pure Retransmission?

There are only two families of remedies for packet loss:

  • Forward Error Correction (FEC) — send redundant data proactively; recovery costs zero extra RTT.
  • Automatic Repeat Request (ARQ / NACKs) — detect loss and request re-send; costs ≥ 1 RTT (NACK + retransmit, both through hostile relay).

For real-time streaming where every millisecond of latency counts, FEC is the primary mechanism. NACKs serve as a backup for the rare multi-loss case.

XOR Parity Groups (K = 4)

Every 4 consecutive frames form a FEC group. The sender computes:

Parity = P₀ ⊕ P₁ ⊕ P₂ ⊕ P₃

If any single frame in the group is lost, the receiver recovers it:

P_missing = Parity ⊕ P_other₁ ⊕ P_other₂ ⊕ P_other₃

5. Jitter Buffer & Playout

The receiver maintains a frame buffer indexed by sequence number. Frames are stored as they arrive (regardless of order) and played out at their deadline:

deadline(i) = t0 + DELAY_MS/1000 + i × 0.020 seconds

Event Loop Design

The receiver uses select() with a 1 ms max timeout to alternate between two tasks:

  1. Receive & process: drain all available packets using MSG_DONTWAIT, store data, update FEC groups, trigger recovery.
  2. Playout: check if any frame deadlines have passed; send available frames to the player, skip misses, advance the playout pointer.

This tight loop ensures sub-millisecond playout precision while remaining responsive to incoming packets for last-moment FEC recovery.

6. NACK Retransmission (Fallback)

Receiver Side — NACK Generation

  • Gap detection: when frame S arrives and frames highest+1 … S−1 are missing, NACKs are sent immediately.
  • Proactive re-NACKs: for upcoming frames whose deadlines are within 70% of DELAY_MS but still >10 ms away, re-NACKs are sent (up to 3 attempts per frame).

Sender Side — Retransmission

  • A ring buffer of 4096 slots stores recent harness frames.
  • On receiving a NACK, the sender looks up the requested seq in the ring buffer and retransmits it as a RETRANSMIT packet.
  • Retransmits use the same wire format as DATA, so the receiver processes them identically (dedup + FEC integration).

RTT Considerations

Both the NACK and the retransmit cross the hostile relay, so minimum RTT is 2 × min_delay (best case: ~20 ms for profile A, ~40 ms for B). NACKs are most useful when DELAY_MS is large; at very low delays, FEC is the only viable recovery mechanism.

7. Sender Architecture Detail

Event Loop

for (;;) {
    select() on { port_47010, port_47004 } with 5ms timeout

    if (frame from harness on 47010):
        1. Save in ring buffer for retransmission
        2. Send DATA packet to relay (47001)
        3. XOR payload into parity accumulator
        4. If group complete (K frames):
           → Send PARITY packet to relay
           → Reset accumulator

    if (NACK from feedback on 47004):
        1. Look up requested seq in ring buffer
        2. If found → send RETRANSMIT packet to relay
}

Key Design Decisions

  • Immediate forwarding: each DATA packet is sent the instant the harness delivers it — no buffering delay at the sender.
  • Parity at group boundary: the parity is sent right after the K-th frame, minimizing the window before the receiver can attempt recovery.
  • Ring buffer validation: stored seq is checked against the request to avoid stale data after wraparound.

10. Design Tradeoffs

DecisionTradeoffRationale
FEC over pure NACK Higher bandwidth vs lower latency Zero-RTT recovery is critical for real-time streaming; NACK RTT often exceeds playout budget
Consecutive grouping (not interleaved) Simpler code vs burst vulnerability Interleaving increases FEC recovery latency; NACKs cover burst-loss gaps
K=4 25% parity overhead vs recovery capability Sweet spot: handles 5% random loss easily, overhead stays at ~1.31×
5-byte wire header Minimal metadata vs protocol flexibility FEC group info derived from seq number (group = seq/K), saving 3+ header bytes per packet
MSG_DONTWAIT drain loop CPU spin vs responsiveness Ensures all queued packets are processed before playout decisions, maximizing FEC recovery window
