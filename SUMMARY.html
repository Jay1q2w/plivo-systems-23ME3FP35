The Flaky Network

Architecture & Design Choices — Real-Time UDP Stream Recovery

1. Problem Statement

A source produces one 160-byte frame every 20 ms (simulating live audio). Frames must cross a hostile UDP relay that drops, delays, reorders, and duplicates packets, and be played out on a steady 20 ms clock at the receiver. We build the sender and receiver programs that sit on either side of this relay.

3. Wire Protocol Design

All sender→receiver packets share a 5-byte header (type + 4-byte big-endian sequence/ID), followed by 160 bytes of payload, totaling 165 bytes. This minimal header was chosen to conserve the 2.0× bandwidth budget.

4. Loss Recovery: Pure XOR FEC (K=2)

Why FEC Over Retransmission?

There are only two families of remedies for packet loss:

  • Forward Error Correction (FEC) — send redundant data proactively; recovery costs zero extra RTT.
  • Automatic Repeat Request (ARQ / NACKs) — detect loss and request re-send; costs ≥ 1 RTT (NACK + retransmit, both through hostile relay).

For real-time streaming where every millisecond of latency counts, FEC is the only viable mechanism to achieve ultra-low playout delays. We completely eliminated NACKs to maximize performance and guarantee fixed overhead.

XOR Parity Groups (K = 2)

Every 2 consecutive frames form a FEC group. The sender computes:

Parity = P₀ ⊕ P₁

If any single frame in the group is lost, the receiver immediately recovers it:

P_missing = Parity ⊕ P_other

5. Sender Architecture Detail

Event Loop

for (;;) {
    read frame from harness on 47010
    1. Send DATA packet to relay (47001)
    2. XOR payload into parity accumulator
    3. If group complete (K=2 frames):
       → Send PARITY packet to relay
       → Reset accumulator
}

Key Design Decisions

  • Immediate forwarding: each DATA packet is sent the instant the harness delivers it — no buffering delay at the sender.
  • Parity at group boundary: the parity is sent right after the K-th frame, minimizing the window before the receiver can attempt recovery.

    if already received → drop (duplicate)

    forward_to_player()

    store parity payload
    try_fec_recovery(group_id)

8. Design Tradeoffs

DecisionTradeoffRationale
Pure FEC without NACKs Lost multi-loss recovery vs fixed overhead/low latency NACKs violate strict delay limits due to full RTT latency. Eliminating them allows driving playout delay to absolute theoretical minimum.
Consecutive grouping (not interleaved) Simpler code vs burst vulnerability Interleaving increases FEC recovery latency, which forces us to increase DELAY_MS. We prioritize lowest delay.
K=2 Highest parity overhead vs lowest recovery latency K=2 is the absolute fastest possible recovery (20ms wait max), easily fitting within the 2.0x overhead budget at 1.55x.
5-byte wire header Minimal metadata vs protocol flexibility FEC group info derived from seq number (group = seq/K), saving 3+ header bytes per packet
